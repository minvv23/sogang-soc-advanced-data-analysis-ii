> **SOC4013-01 고급데이터분석 Ⅱ(캡스톤디자인)** | 2026년 봄학기 첫 수업

## 1. 이 수업이 직시하는 두 가지 딜레마

### 딜레마 1: 학계 vs. 산업

- 학계와 산업을 단순히 대립시키는 건 흔한 구도지만, 사실 이 둘을 제대로 이해하려면 각각을 하나의 **업계(industry)**로 봐야 한다
  - 업계에는 고유한 KPI가 있고, 그 KPI가 행동 양식을 결정한다

- **학계라는 업계**
  - 학계 연구는 science에서 출발하지만, 학자들의 실질적 KPI는 **논문 출판**이다
  - 어느 저널에 몇 편을 냈는가가 임용·승진·연구비의 기준이 된다
  - 탑저널(top journal) 한 편이냐, SSCI 등재지 여러 편이냐 하는 선택도 소속 기관의 평가 기준에 따라 달라진다
  - 이 구조는 연구 방식에도 영향을 미친다
    - **robustness check**: 분석 결과가 표본, 모형 specification, 변수 측정 방식 등을 바꿔도 일관되게 유지되는지 검증하는 과정. 저널 심사자들이 거의 의무적으로 요구하는 절차가 됐다
    - 과학적으로 당연히 중요한 작업이지만, 동시에 "저널에 통과하기 위한 관례"이기도 하다
    - 학계의 방법론적 엄밀성 중 상당 부분은 science 그 자체라기보다, 그 과학을 특정 업계 기준에 맞게 포장하는 방식이다

- **산업계라는 업계**
  - 산업 현장에서는 의외로 복잡한 데이터 분석을 잘 쓰지 않는다
  - 이유는 단순하다: **ROI가 안 나온다**
  - 정교한 인과추론 모형을 짜는 데 드는 시간과 비용보다, 빠른 A/B 테스팅이 훨씬 실용적이다
  - 실제로 dynamic pricing, 광고 추천, 이탈 예측처럼 산업의 핵심 데이터 문제들은 대부분 A/B 테스트로 충분히 해결된다
  - 복잡한 DID나 IV를 쓸 일은 생각보다 드물다

- **그래서 이 수업이 택하는 방향**
  - 학계도 산업계도 각자의 인센티브 구조 안에서 움직이는 업계라는 사실을 아는 것이 출발점이다
  - 이를 모르면 학계 방식을 맹목적으로 따르거나, 산업 현장의 단순함을 과소평가하게 된다
  - 이 수업은 두 업계의 논리를 이해한 위에서, 각각의 장점—학계의 통계적 엄밀성, 산업계의 실용적 문제 설정—을 취한다

### 딜레마 2: AI 시대의 데이터 분석 수업의 의미

- ChatGPT, Claude, Gemini는 이미 회귀분석 코드를 짜주고, 계수를 해석해주고, 논문 초안까지 써준다
- 2025년부터는 여기서 한 단계 더 나아간 **에이전틱 코딩 도구**들이 등장했다
  - 채팅창에 답해주는 수준이 아니라, 터미널에서 직접 코드를 실행하고, 오류를 보고 스스로 고치고, 분석 파이프라인 전체를 자율적으로 완주한다
  - 실무에서는 `CLAUDE.md`, `skills.md`, `AGENTS.md` 같은 파일을 프로젝트 루트에 두는 관행이 생겼다
  - 여기에 "무엇을 분석하는지, 데이터는 어디 있는지, 어떤 방식으로 분석해야 하는지"를 적어두면, 에이전트는 이 파일을 읽고 사람의 개입 없이 파이프라인 전체를 실행한다
  - **누군가의 분석 업무가 `skills.md` 한 장으로 기술될 수 있다면, 그 업무는 이미 자동화 가능하다**

- 이미 실제로 일어나고 있는 일이다
  - [Reddit 경험담 분석](https://www.restless-brain.com/p/ai-is-replacing-real-jobs-right-now): 13년 경력 데이터 사이언티스트가 "Copilot이 이렇게 하라고 했다"는 이유로 해고됨. 6년 경력 분석가 7명이 내부 AI 유지 담당자 1명으로 대체됨
  - [Harvard](https://careerservices.fas.harvard.edu/blog/2025/09/25/will-data-analysts-be-replaced-by-ai/) · [HBR](https://hbr.org/2026/01/companies-are-laying-off-workers-because-of-ais-potential-not-its-performance): entry-level 분석직이 먼저 사라지고 있다 — 주니어를 키우는 게 아니라 AI에게 시키고 시니어가 감독하는 구조
  - [Brookings](https://www.brookings.edu/articles/the-train-has-left-the-station-agentic-ai-and-the-future-of-social-science-research/): "기차는 이미 출발했다" — 에이전틱 AI가 강건성 검정·재현·분석 계획 수립을 대신하면서 동일한 연구 결과를 더 적은 인원으로 낼 수 있게 됐다

- **일반론적 교훈: 분석 기술 자체는 더 이상 희소하지 않다**
  - 회귀분석을 돌릴 줄 안다는 것, 패널 데이터를 다룰 줄 안다는 것—이것만으로는 충분하지 않다
  - AI가 못 하는 것—어떤 질문을 던질지 알고, 데이터를 직접 만들어낼 수 있고, 분석 결과의 의미를 맥락 속에서 판단하는 것—을 할 수 있어야 한다
  - 이 수업의 답은 그 영역에 집중하는 것이다

## 2. "딸깍"으로 대체 당하는 분석가 vs. 그렇지 않은 분석가

### AI가 잘 하는 것 (= "딸깍" 가능한 영역)

- 표준적인 분석 코드 작성 (회귀분석, 시각화, 전처리)
- 이미 정의된 문제에 대한 분류(classification) 수행
- 교과서에 있는 개념 설명
- 결과 요약 및 보고서 초안 작성
- **2025년부터: 에이전틱 코딩 도구** — 코드를 짜주는 게 아니라 분석 자체를 수행
  - **[Claude Code](https://github.com/anthropics/claude-code)** (Anthropic): 터미널 기반. 코드베이스 전체를 이해하고 자연어 명령으로 분석 작업 수행. 2025년 5월 출시 후 연 환산 매출 $1B 돌파
  - **[Codex CLI](https://github.com/openai/codex)** (OpenAI): 오픈소스, 터미널 기반. Python 샌드박스에서 데이터 분석·파일 변환·반복 디버깅 수행. 멀티 에이전트 병렬 실행 지원
  - **[Gemini CLI](https://github.com/google-gemini/gemini-cli)** (Google): 오픈소스. 100만 토큰 컨텍스트 윈도우, Google Search 연동, MCP 지원. 개인 Google 계정으로 무료 사용 가능 (일 1,000회)

### AI가 잘 못 하는 것 (= 인간 분석가의 가치 영역)

- **① 첫 질문을 던지는 것**
  - AI는 주어진 질문에 답하는 데는 매우 능숙하다. 하지만 "무엇을 물어야 하는가"는 여전히 인간의 몫이다
  - "용산 이전이 교통에 영향을 줬는가"라는 질문 자체를 떠올리려면, 당시의 정치·사회적 맥락과 데이터 분석의 가능성을 동시에 알고 있어야 한다
  - 어떤 현상이 분석할 가치가 있는지, 어떤 데이터가 존재하는지를 아는 것은 도메인 지식과 현장 감각 없이는 불가능하다

- **② 몸뚱이가 필요한 일**
  - 정보공개청구: 공문 작성, 담당 부서 파악, 재청구, 이의신청—이 과정은 실제 사람이 제도 안에서 움직여야 한다
  - 설문 설계: 응답자를 모집하고, 실제 현장에서 테스트하고, 문항을 다듬는 과정이 수반된다
  - AI가 분석을 도와줄 수는 있어도, 분석할 데이터를 세상에서 끌어오는 일은 여전히 인간의 몫이다

## 5. 수업 운영에 관한 몇 가지 원칙

### AI 활용에 대하여

이 수업은 AI 사용을 금지하지 않는다. 오히려 적극 권장한다. 단, 다음을 구분할 줄 알아야 한다.

- AI가 알려주는 것을 **그대로 믿는** 것과 **비판적으로 검토하는** 것
- AI가 코드를 짜준 것을 **그냥 실행하는** 것과 **왜 그렇게 짰는지 이해하는** 것
- AI가 결과를 해석해준 것을 **보고서에 붙이는** 것과 **맥락에 맞게 판단하는** 것

ChatGPT, Claude, Gemini 중 2개 이상의 유료 플랜을 권장하는 이유도 여기에 있다. 여러 모델을 교차 검증하는 습관을 들이는 것이 목표다.

### 실습 환경

- **OS**: Windows Subsystem for Linux (WSL) 또는 macOS
- **언어**: Python
- 별도 프로그래밍 수업은 없음. 기본 Python 경험 전제

### 왜 WSL인가 — 그냥 윈도우에서 하면 안 되나?

데이터 분석 실무와 연구는 압도적으로 **Linux/Unix 환경**을 전제로 돌아간다. 그 이유는 단순하다.

- 대부분의 오픈소스 라이브러리, 크롤러, 분석 파이프라인은 Linux에서 개발되고 테스트된다. 윈도우에서는 설치 자체가 안 되거나, 되더라도 경로 문제·인코딩 문제·권한 문제가 수시로 터진다.
- 서버(클라우드 포함)는 거의 전부 Linux다. 로컬에서 윈도우로 개발하고 서버에 올리면 환경이 달라 안 돌아가는 경우가 생긴다.
macOS는 Unix 기반이라 대부분 그냥 동작한다. 윈도우 사용자라면 WSL2를 설치하면 사실상 동일한 환경을 쓸 수 있다.

### WSL1 vs WSL2

WSL에는 두 버전이 있다. 반드시 **WSL2**를 써야 한다.

| | WSL1 | WSL2 |
|--|------|------|
| 아키텍처 | Linux 시스템 콜을 윈도우 API로 변환 (번역 레이어) | 실제 Linux 커널을 경량 VM으로 실행 |
| 파일시스템 I/O | 빠름 (윈도우 파일에 접근 시) | Linux 파일시스템 내에서는 훨씬 빠름 |
| 호환성 | 일부 시스템 콜 미지원 → 라이브러리 설치 실패 잦음 | 완전한 Linux 커널 → 호환성 문제 거의 없음 |
| Docker | 지원 안 됨 | 지원 |

데이터 분석에서 쓰는 라이브러리들(특히 크롤러, 바이너리 의존성 있는 패키지들)이 WSL1에서는 설치 자체가 안 되는 경우가 빈번하다. WSL2는 실제 Linux 커널이 올라가기 때문에 이런 문제가 없다.

### WSL2 설치 방법 (Windows 사용자)

**1단계: PowerShell을 관리자 권한으로 열고 아래 명령어 실행**

```powershell
wsl --install
```

Windows 10 버전 2004(빌드 19041) 이상에서는 `wsl --install` 실행 시 자동으로 WSL2가 기본값으로 설치된다. 설치 후 재부팅.

**1-1단계: WSL2인지 반드시 확인**

재부팅 후 PowerShell에서:

```powershell
wsl --list --verbose
```

출력 예시:
```
  NAME      STATE           VERSION
* Ubuntu    Running         2
```

`VERSION` 열이 **2**이면 정상. 만약 1이라면:

```powershell
wsl --set-version Ubuntu 2
```

앞으로 새로 설치하는 배포판도 WSL2로 강제하려면:

```powershell
wsl --set-default-version 2
```

**2단계: Ubuntu 실행 후 사용자 이름·비밀번호 설정**

시작 메뉴에서 "Ubuntu" 실행 → username과 password 입력 (비밀번호는 입력해도 화면에 표시 안 됨, 정상)

**3단계: 패키지 업데이트**

```bash
sudo apt update && sudo apt upgrade -y
```

**4단계: Anaconda 설치**

Anaconda 공식 사이트에서 Linux용 설치 스크립트 링크를 복사한 뒤 WSL 터미널에서 실행한다.

```bash
wget https://repo.anaconda.com/archive/Anaconda3-2024.10-1-Linux-x86_64.sh
bash Anaconda3-2024.10-1-Linux-x86_64.sh
```

설치 완료 후 터미널을 재시작하거나 `source ~/.bashrc` 실행. 이후 `conda` 명령어로 패키지 관리.

```bash
conda --version  # 설치 확인
```

> **참고**: WSL에서 작업한 파일은 윈도우 파일 탐색기에서 `\\wsl$\Ubuntu\` 경로로 접근할 수 있다. 반대로 윈도우의 C 드라이브는 WSL에서 `/mnt/c/`로 접근된다.
